# Query planning

You write a query saying what you want. The database decides how to run it. That decision is the query plan, and the same query can be run in very different ways with very different costs.

Two decisions matter here. First, whether to use an index or just scan the whole table: an index is fast when few rows match, but once a large fraction of the table matches, reading everything in one pass can be cheaper. Second, when a query joins several tables, the order to join them: joining the small, filtered table first keeps the intermediate result small, while a bad order can blow it up.
