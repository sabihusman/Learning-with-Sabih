# Tokenization

Before a language model can do anything with text, the text has to be turned into numbers. That first step is tokenization: the input is chopped into pieces called tokens, and each token is mapped to an integer ID. The model only ever sees those IDs. It does not read words, and it does not read letters. It reads tokens.

## Tokens are not words

A token is not the same as a word. Some common words are a single token, but longer or rarer words get split into smaller subword pieces. A word like "apple" might be one token, while "unbreakable" is split into "un", "break", and "able", three separate tokens. Splitting this way lets a fixed vocabulary cover almost any word, including ones it has never seen, by assembling them from familiar pieces.

Because of this, the number of tokens rarely matches the number of words. In the figure below, edit the sentence and watch the token count change independently of the word count.

## Why this matters

Since the model reads tokens, not letters, it has no direct view of the characters inside a token. When a whole word like "strawberry" becomes a single token, the model sees one opaque unit, not s-t-r-a-w-b-e-r-r-y. This is a big part of why models can struggle with spelling, rhyming, or counting the letters in a word: the letters were swallowed into a token the model treats as indivisible. Words that split into several tokens expose more of their structure, which is one reason behavior can differ between a word the model sees whole and one it sees in pieces.

## What the demo shows

Each colored block is one token, labeled with its integer ID. Common words stay whole, while longer words break into subword pieces, with the original word shown beneath. The readout compares tokens to words and characters so you can see they are not the same. The tokenizer here is a simplified, rule-based illustration with a tiny fixed vocabulary, not a real model's learned tokenizer, but the idea is exactly what real models do: read text as a sequence of token IDs.
