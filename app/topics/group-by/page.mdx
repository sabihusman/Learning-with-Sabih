# GROUP BY and aggregation

`GROUP BY` answers questions about groups rather than individual rows. Instead of one row per session, you ask for one row per country, or one row per plan, and a number that summarizes each group. To do that, the database takes all the rows that share a grouping value, folds them together, and returns a single summary row for each group. The detail rows disappear into their totals.

This is the opposite of a window function. A window function computes across rows but keeps every row visible. `GROUP BY` collapses them: many rows in, one row per group out.

## Choosing the aggregate

The summary value comes from an aggregate function applied to each group. `COUNT(*)` counts the rows in the group. `SUM(events)` adds up a column across the group, and `AVG(events)` averages it. The figure groups a slice of sessions, where each row carries the user's country and plan and the number of events recorded in that session, so the aggregate has something real to add up.

## COUNT(\*) versus COUNT(DISTINCT)

`COUNT(*)` and `COUNT(DISTINCT user_id)` look similar and often disagree. `COUNT(*)` counts every row. `COUNT(DISTINCT user_id)` counts how many different users appear, so if one user has several sessions in a group, those extra rows fold into one. In the figure, the repeat rows fade out when you pick the distinct count, and the group's number drops to the number of unique users. Reaching for the wrong one is a common way to over- or under-count.

## WHERE versus HAVING

Filtering happens at two different moments. `WHERE` filters rows before grouping, deciding which rows are even eligible. `HAVING` filters groups after aggregating, deciding which summary rows survive based on their aggregate value. The slider in the figure is a `HAVING COUNT(*) > N` filter: raise it and whole groups drop out once they are too small, while the rows inside the surviving groups are untouched. You cannot do that with `WHERE`, because the group counts do not exist until after grouping.
